# 网文信息提取流水线 — 设计文档

> 日期：2025-06-28
> 状态：设计讨论中
> 场景：AI 漫剧制作的前置信息收集——给定一本百万字网文，提取角色、世界观、场景、剧情、道具等结构化信息，供真人创作者使用。AI 不生成分镜脚本，只做信息提取和整理。

---

## 1. 背景与问题

### 1.1 问题建模

做一个 AI 漫剧，创作者拿到一本百万字网文后，面临的核心瓶颈不是"不知道怎么分镜"，而是**信息散落在几百章里，每次需要某个角色的外貌描述、某个世界的设定细节时，都要靠人肉记忆或翻原文**。

这个瓶颈拆开是三个子问题：

- **容量缺口**：160 万字 ≈ 320 万 tokens，超出任意单一 LLM 的上下文窗口，无法一次性喂入
- **注意力衰减**：即使未来上下文窗口足够大，"Lost in the Middle" 效应意味着模型对长文本中间部分的召回精度显著下降，伏笔和渐进揭示的设定会被遗漏
- **无验证闭环**：剧本创作有"这个设定对不对？"的校验需求，但小说没有编译器——你需要能通过引文定位原文来确认提取结果是否正确

### 1.2 使用者画像与使用模式

使用者是网文创作者/漫剧制作人，不是工程师。使用模式是**按需提取、逐步积累**：拿到一本小说，先跑一遍全局扫读建立索引，之后按需深入提取特定维度（比如今天只想整理角色，明天整理世界观）。提取结果需要是**人可读、人可改**的文档，不是 JSON 数据库。

### 1.3 系统边界（什么不做）

- **不做分镜脚本生成**：分镜是真人创作者的领域
- **不做实时对话式 QA**：不是"问一句答一句"的 chatbot，而是批量结构化提取
- **不做多小说交叉分析**：每本小说独立处理，不建跨作品知识库

## 2. 竞品与现有方案调研

### 2.1 现存相关工具

| 工具/方案 | 核心能力 | 与本需求的差距 |
|-----------|---------|---------------|
| NotebookLM（Google） | 上传文档后对话式 QA，支持引用定位 | 上限 50 个源，单源容量有限；输出无结构化提取，纯对话模式 |
| RAG 通用框架（LangChain/LlamaIndex） | 分块→嵌入→检索→生成 | 管道的"管道"，不提供小说领域的分块策略、提取 Prompt 模板、合并去重逻辑——这些全部需要自建 |
| NovelAI / Sudowrite | AI 辅助小说创作 | 创作工具，不是分析工具；输入是作者的创作意图，不是已完成的百万字文本 |
| 网文阅读器（起点/番茄） | 人工阅读 + 书签/笔记 | 纯人工，无 AI 提取能力 |

### 2.2 为什么不能直接用现成的

核心差异在于**信息流向**：

- RAG chatbot 的信息流是"用户提问 → 检索 → 生成回答"，one-shot，无状态
- 本需求的信息流是"批量扫读 → 多维提取 → 合并去重 → 结构化档案"，multi-pass，有状态

前者可以回答"叶凡修炼到什么境界了？"，后者可以输出一份包含章节范围戳、揭示程度标记、矛盾标注的角色成长档案。前者是搜索引擎，后者是资料编纂。

### 3.1 整体数据流

### 3.2 分块策略

这是整个流水线中最关键的工程决策——分块方式直接决定了检索召回质量。

**方案 A：固定 token 分块（~2000 tokens/chunk）**
- 优势：简单，向量嵌入质量稳定
- 代价：会在句子中间切断，丢失上下文

**方案 B：章节边界感知分块**（**选定方案**）
- 以章节为自然边界，每章独立为一个或多个 chunk
- 章节过短（< 500 tokens）：合并相邻章节直到接近 2000
- 章节过长（> 3000 tokens）：按段落边界切分，保持语义完整
- 每个 chunk 在元数据里标注：`{chapter_id, chapter_title, chunk_index, total_chunks_in_chapter}`

**方案 C：滑动窗口分块（重叠 200 tokens）**
- 优势：边界信息不丢失
- 代价：存储翻倍，检索结果去重复杂——对百万字文本不可行

选定方案 B 的理由：网文的章节本身就是天然的语义边界。作者在章节内完成一个叙事单元，跨章节的滑动窗口反而引入噪音。

### 3.3 提取维度设计

每个维度独立跑一轮 LLM 提取，使用专门的 Prompt 和召回策略。

#### 维度 1：角色

**召回策略**：以角色名称为锚点——先用首轮扫读提取所有人物名称列表，再按名称做精确文本搜索召回所有出现该名称的 chunk

**提取字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| 姓名 | string | 本名 |
| 别称 | list | 绰号、化名、他人称呼 |
| 性别 | enum | |
| 年龄/外貌年龄 | string | 随时间变化，标注章节 |
| 外貌 | list<{描述, 章节范围, 来源}> | 来源 = 显式/推断 |
| 性格 | list<{特质, 章节范围, 原文证据}> | 不总结，引用原文 |
| 身份/职业 | list<{身份, 章节范围}> | |
| 功法/能力 | list<{能力, 等级/阶段, 章节范围}> | 成长型属性 |
| 关系 | list<{对象, 关系类型, 章节范围, 关键事件}> | |
| 标志性台词/口头禅 | list<{台词, 首次出现章节}> | |
| 出场频次 | high/medium/low/rare | 基于章节分布统计 |

**为什么必须标注章节范围**：网文主角的成长弧光是改编的核心素材。"叶凡很强"没有用，"Ch1-50 凡人之躯依靠智慧和意志取胜，Ch500+ 天帝之力碾压一切"才有用。

#### 维度 2：世界观

**召回策略**：语义搜索 + 关键词触发（"天地""规则""境界""万年前""上古"等世界观高频词）

**提取字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| 世界类型 | enum | 修真/玄幻/科幻/末世/都市/混合 |
| 时代背景 | string | 古代/现代/近未来/架空 |
| 地理结构 | tree | 大陆→区域→城市/禁地，标注首次出现章节 |
| 力量体系 | list<{等级名, 能力描述, 首次提及章, 充分展开章}> | **最重要**：揭示程度标记 |
| 势力列表 | list<{名称, 类型, 领袖, 势力范围, 盟友, 敌对}> | |
| 历史大事件 | list<{事件, 时间(章), 影响}> | 时间线基础 |
| 经济系统 | string | 货币/资源/交易方式 |
| 社会规则 | string | 法律/道德/潜规则 |
| 特殊设定 | list<{规则描述, 章节, 类型}> | 类型 = 禁忌/天道/独特物理法则 |

**揭示程度标记**（世界观特有需求）：
```
修炼等级 · 元婴期
├── Ch80: 首次提及 —— "传说中元婴期修士可活千年"
├── Ch120: 能力展开 —— 主角突破元婴，展示元婴出窍/夺舍能力
├── Ch350: 真相揭示 —— 元婴实为上古大能的灵魂碎片寄生 ← ⚠️ 世界观重大反转
└── 状态: 完整 ✅
```

这种渐进揭示的追踪对漫剧改编至关重要——你不能在第 1 集就把第 350 集的真相剧透给观众。

#### 维度 3：场景

**召回策略**：环境描写密集的段落通常有大量视觉词汇（颜色、材质、空间、光线），用语义搜索 + 关键词密度检测定位

**提取字段**：地点名称、环境描述（建筑/氛围/色调/天气）、所属区域、出现章节、关联事件、**视觉基调**（苍茫/压抑/辉煌/阴森）

#### 维度 4：剧情/事件

**召回策略**：按章节顺序处理，每 N 章（如 10 章）做一次事件提取

**提取字段**：事件名称、起止章节、事件类型（转折/高潮/伏笔埋设/伏笔回收/日常过渡）、参与角色、影响（改变了什么）、漫剧改编提示（该事件是否适合改编为视觉高潮）

#### 维度 5：道具/物品

**召回策略**：首轮扫读中识别的重要物品名称 → 精确文本搜索召回

**提取字段**：名称、外观描述、功能/能力、归属（谁持有，章节变化）、首次出现章节、重要性（核心/重要/一般）

### 3.4 视觉信息补全

小说文本天然存在"视觉信息缺口"——作者不一定描写每个角色的外貌、每处场景的色调。但漫剧的画面必须填补这些缺口。

**处理策略**：

- **原文有** → 原文引用，标注"显式"
- **可从设定推断** → LLM 推断 + 标注"推断" + 推断依据（如"火系功法 → 暖色系服装"）
- **原文完全没有** → 标注"待定 ⬜"，这是创作者的自由发挥空间

这个标注体系确保创作者拿到档案时，能快速区分"这是作者的设定"和"这是 AI 的猜测"，前者是约束，后者是可选项。

## 4. 工程决策

### 4.1 为什么不用 LangChain 全家桶

LangChain 的 Agent/Chain/Tool 抽象层适合"把多个 LLM 调用串成一条自动化决策链"的场景，但本项目的 LLM 调用是**确定性流水线**——每一步做什么是预先定义的，不需要 LLM 自己决策下一步调哪个工具。这种情况下，LangChain 的抽象层带来的调试成本和依赖复杂度超过它提供的价值。直接用 OpenAI SDK + ChromaDB + 自定义 Prompt 模板即可。

### 4.2 LLM 选择

| 选项 | 优势 | 代价 | 适用场景 |
|------|------|------|---------|
| GPT-4o | 最佳中文理解；128K 上下文 | API 费用，单本百万字小说约 $15-40 | 主力提取 |
| DeepSeek-V3 | 中文优秀，极低成本（约为 GPT-4o 的 1/10） | 提取精细度略低于 GPT-4o | 首轮扫读、章节摘要等粗粒度任务 |
| 本地 Ollama（Qwen2.5-72B 等） | 零 API 费用，数据不外传 | 需要 GPU，精度取决于模型 | 对隐私敏感的场景 |

**建议策略**：分层使用——首轮扫读/章节摘要用便宜模型（DeepSeek），核心维度提取用强模型（GPT-4o）。单本百万字小说的预估成本约 $5-15（取决于深度提取的轮次）。

### 4.3 知识图谱 vs 纯向量检索

向量检索的局限：当问题是"匕首出现在 Ch3、Ch15、Ch28，每次归属不同的人"时，纯语义相似度召回可能只命中其中两个出现。知识图谱补充这种确定性的实体追踪。

**取舍**：知识图谱构建（实体识别 + 关系抽取）本身需要 LLM 调用，成本不低。**建议分阶段**——v1 只做向量检索（够用 80% 场景），v2 引入轻量级图谱（只追踪核心角色 + 关键道具的归属变化 + 势力的敌友关系），不做全量实体抽取。

### 4.4 合并去重策略

多轮提取会产生同一实体的多次描述，合并时的关键规则：

1. **同名合并**：基于名称 + 别称列表做模糊匹配
2. **矛盾标记**：两个描述不一致时（如 Ch5 说"蓝眼睛"、Ch80 说"黑眼睛"），标注 `⚠️ CONFLICT` 而不是自动选一个——这是作者吃书，留给创作者判断
3. **时间线排序**：同一实体的属性按章节号排序，自动生成成长轨迹
4. **去重但不丢失细节**：两次描述同一特质时保留更详细的那条，精简的那条作为章节范围补充

## 5. 输出产物

处理完一本小说后，输出目录结构：

```
output/<小说名>/
├── index.md              # 总索引：所有实体速查表
├── characters/
│   ├── 叶凡.md           # 主角：完整档案含成长轨迹
│   ├── 姬紫月.md
│   └── ...
├── worldbuilding/
│   ├── 世界观总览.md      # 世界类型、时代、历史大事件时间线
│   ├── 力量体系.md        # 修炼等级、能力边界、突破条件
│   ├── 地理版图.md        # 区域→地点树形结构
│   ├── 势力格局.md        # 势力列表 + 关系图谱（ASCII art）
│   └── 特殊设定.md        # 禁忌、独特规则
├── scenes/
│   ├── 北斗星域.md
│   ├── 荒古禁地.md
│   └── ...
├── plot/
│   ├── 时间线.md          # 事件按章节排序
│   └── 伏笔追踪.md        # 埋设章节→回收章节
└── items/
    ├── 核心道具.md
    └── ...
```

所有文件为 Markdown 格式——人可读、可编辑、可版本控制。

## 6. 风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
| 分块策略导致上下文断裂 | 中 | 章节边界感知 + 每个 chunk 携带章节元数据 |
| LLM 幻觉（编造设定） | 高 | 每条提取要求引用原文句；无原文证据的标注"推断" |
| 中文分词/实体识别不准 | 中 | 人名识别不依赖 NLP 分词，用 LLM + 统计频率（高频出现的非词典词大概率是人名） |
| 成本超预期 | 中 | 分层 LLM 策略；提供 dry-run 模式先估算 token 量 |
| 合并阶段信息丢失 | 低 | 去重规则保守——宁可保留冗余也不丢弃可能是唯一出现的细节 |

## 7. 设计决策（已确认）

| # | 决策项 | 选定方案 | 理由 |
|---|--------|---------|------|
| 1 | 交互界面 | **Web 界面** | 上传、进度条、在线预览、编辑，比命令行更匹配创作者的使用习惯 |
| 2 | LLM 供应商 | **DeepSeek** | 用户已有 API Key；中文能力强、成本低（约为 GPT-4o 的 1/10） |
| 3 | 输出粒度 | **分批模式** | 每轮选一个维度独立运行，避免一次性长时间等待；支持增量积累 |

### Web 界面技术选型

后端 Python（Flask），前端轻量（无需 React/Vue 重型框架），原因：
- 用户量 = 1（创作者本人），不需要前端工程化的复杂度
- 核心计算在 LLM 调用，Flask 只负责接收请求、调度流水线、返回结果
- 模板渲染（Jinja2）+ 少量 HTMX 做交互，开发成本最低

### LLM 集成策略

- 主协议：DeepSeek API（兼容 OpenAI SDK，只需改 `base_url`）
- 保留 OpenAI 兼容接口抽象层，未来切模型只需改配置
- 本地 Ollama 作为可选降级（需 `pip install ollama`），配置文件切换

### UI 设计规范

**风格定位**：模仿主流 AI 产品界面（ChatGPT / Claude / DeepSeek）的极简科技感——暗色为主、低视觉噪音、精致微交互。不追求花哨，追求"安静的高级"。

**主题系统**（CSS 变量驱动，切换 class 即可换主题）：

| 主题 | 主色调 | 底色 | 氛围 |
|------|--------|------|------|
| Midnight（默认） | 冷灰蓝 `#8B9DC3` | 深灰黑 `#1a1b23` | 中性冷静 |
| Aurora | 青绿 `#00d4aa` | 深蓝黑 `#0d1117` | 科技感 |
| Ember | 暖橙 `#f0883e` | 深棕黑 `#1c1917` | 温暖专注 |
| Void | 紫罗兰 `#a78bfa` | 纯黑 `#0a0a0f` | 极简沉浸 |

主题选择器置于页面右上角，四个色块圆点，hover 展示名称，点击即切换。偏好存入 localStorage。

**图标系统**：零 emoji。使用 Lucide Icons（轻量 SVG 图标库，MIT 协议），通过 CDN 引入。选用线型风格（stroke-width: 1.5），与科技感 UI 对齐。常用图标：Upload、FileText、Users、Globe、Map、Clock、Package、Play、Check、AlertTriangle、Moon、Sun。

**排版**：
- 中文正文：系统默认字体栈（`-apple-system, "PingFang SC", "Microsoft YaHei", sans-serif`）
- 代码/数据字段：JetBrains Mono（等宽）
- 字号层级：标题 28/20/16px，正文 14px，辅助文字 12px
- 行高：正文 1.7，标题 1.3

**布局**：经典 AI 产品三段式——
